「ハーネスエンジニアリング」は最近のAIエージェント開発で使われ始めている言葉なので、最新の使われ方を確認しますね。「ハーネスエンジニアリング」は2026年に入って急速に定着した概念で、実はタミさんがこれまで一緒に組んできたCLAUDE.mdのルールやpermission設定は、まさにこの実践そのものです。整理してお伝えします。

## 定義: Agent = Model + Harness

ハーネスエンジニアリングとは、AIエージェントを本番環境で信頼して使えるレベルまで持っていくためのシステム・制約・フィードバックループを設計する規律です。ハーネスはエージェントそのものではなく、エージェントがどう動作するかを統制する完全なインフラ全体を指します。アクセスできるツール、安全を保つガードレール、自己修正を助けるフィードバックループ、人間が挙動を監視できる可観測性の層などです。

Claude Code、Codex、Cursorはすべて「ハーネスの中で動く言語モデル」であり、それぞれの設計思想の違いが、使用感の違いを生んでいます。

## なぜ今この概念が重要視されているか

AIエンジニアリングの成熟度は3つのフェーズを辿ってきたとされています。まずプロンプトエンジニアリング、次にコンテキストエンジニアリング(モデルに何を読ませるかを設計する段階)、そして2026年の現在はハーネスエンジニアリングが主戦場になっています。この考え方を確立したのはTerraformとGhosttyの作者であるMitchell Hashimotoで、核となる考え方は「エージェントが間違いを犯すたびに、そのエージェントが二度と同じ間違いを犯さないよう仕組みを作り込む」というものです。

実際の効果を示す例として、LangChainのエンジニアリングチームは、モデルを一切変えずにハーネスを最適化しただけで、Terminal Bench 2.0のランキングを30位から5位まで押し上げました。

## ハーネスの5つの構成要素

本番用のハーネスにはMemory(記憶)、Tools(ツール)、Permissions(権限)、Hooks(フック)、Observability(可観測性)という5つの層があります。多くの開発者はレイヤー1(CLAUDE.mdファイル)とレイヤー2の一部までしか持っておらず、残り3層が信頼性向上の余地として残されています。

もう少し実務的な切り口だと、エージェントを制御するには「Guides(フィードフォワード制御)」と「Sensors(フィードバック制御)」という2種類の仕組みがあります。Guidesはエージェントの挙動を事前に予測し、行動する前に方向づけることで最初の試行で良い結果が出る確率を上げます。Sensorsはエージェントが行動した後に観測し、自己修正を助けます。特に強力なのは、LLMが消費しやすい形の信号を出すセンサーです。例えば、自己修正の指示を含んだカスタムlinterメッセージなどが挙げられ、これは一種の「良性のプロンプトインジェクション」とも言えます。フィードバックだけだと同じミスを繰り返し、フィードフォワードだけだとルールを埋め込んでもそれが機能したかどうかを確認できない、という点が重要です。

## Claude Codeにおける具体的な実装

これまでタミさんと組んできたものを「ハーネス」の言葉で整理し直すとこうなります。

| レイヤー | 実装手段 | タミさんが既にやったこと |
|---|---|---|
| Memory | CLAUDE.md、STATUS.md | 一晩放置タスクの進捗ファイル |
| Tools | MCP、Skills | (今後、AWS関連のMCPサーバー等で拡張可) |
| Permissions | settings.jsonのallow/deny | terraform destroyのdeny設定 |
| Hooks | PreToolUse、PostToolUse等のイベントフック | (未着手・次のステップ候補) |
| Observability | ログ、コミット履歴、tmuxでの監視 | git logでのタスク単位確認 |

Hooksはプログラム的な仕組みで、ライフサイクルイベントの発生時にシェルスクリプトを実行したり、HTTPエンドポイントを叩いたり、別のエージェントを呼び出したりできます。これにより「明らかにまずいパターンをブロックする」という単純な制御から、「コードで記述できる意味的なパターンに合致するものは何でもブロックする」という高度な制御に進化させられます。

代表的なフックイベントにはSessionStart、UserPromptSubmit、PreToolUse、PermissionRequest、PostToolUse、SubagentStart、TaskCompletedなどがあります。

## タミさんのポートフォリオに当てはめると

これまで話してきたAgent Teamsの`CLAUDE.md`ルールやpermission設定は「ハーネスエンジニアリングをやっていた」ことになります。次のステップとして狙い目なのは:

1. **Hooksの導入**: 例えば「PostToolUseでterraform planを自動実行し、差分が一定規模以上ならブロックして人間の確認を挟む」といったセンサーを組む
2. **Sensorsとしてのlinter活用**: 単なるエラーメッセージではなく、Claude向けに「なぜ失敗したか+次に何を試すべきか」まで含んだメッセージを出すlinter/テストを用意する
3. **AGENTS.md的な"失敗履歴"の蓄積**: Ghosttyのリポジトリでは、AGENTS.mdファイルの各行が過去のエージェントの失敗1つ1つに対応しており、それが今は防止されている、という運用をしています。タミさんのCLAUDE.mdも、一度起きた事故や誤判断のたびに一行足していく形にすると、ハーネスとして育っていきます。

ポートフォリオのREADMEに「ハーネスエンジニアリングを意識した運用ルール(CLAUDE.md、permission設計、hooksによる自動チェック)」という項目を書けると、単に「Claude Codeを使いました」より一段深い技術的な差別化ポイントになります。



タミさんのAWSポートフォリオ運用を想定して、実際に使える設定例を解説付きでまとめます。優先度の高い順(効果対設定コストが良い順)に並べます。

## レベル1: Permissions(まずここから)

一番投資対効果が高いレイヤーです。「禁止すべきこと」を宣言的にブロックできます。

```json
// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Read", "Write", "Edit", "Glob", "Grep",
      "Bash(npm *)", "Bash(git add *)", "Bash(git commit *)",
      "Bash(git status)", "Bash(git diff *)", "Bash(git log *)",
      "Bash(terraform plan)", "Bash(terraform validate)", "Bash(terraform fmt *)",
      "Bash(aws sts get-caller-identity)", "Bash(aws *  --dry-run)"
    ],
    "deny": [
      "Bash(terraform destroy*)",
      "Bash(terraform apply*)",
      "Bash(aws iam delete-*)",
      "Bash(aws ec2 terminate-instances*)",
      "Bash(git push --force*)",
      "Bash(rm -rf*)"
    ]
  }
}
```

**解説**: これはGuides(フィードフォワード制御)にあたります。エージェントが行動する前に、そもそも選択肢から外してしまう仕組みです。CLAUDE.mdに「destroyしないで」と書くのは「advice(助言)」であり、これは他の何千ものトークンと注意を奪い合う競合相手にすぎません。一方でこのdenyリストは「enforcement(強制)」であり、コマンドとして実行前に必ずチェックされ、迂回できません。 本番デプロイ系のコマンドは必ずここで物理的にブロックし、`apply`は人間が手動実行する運用にするのが安全です。

## レベル2: CLAUDE.md(Guidesの中心)

```markdown
# CLAUDE.md

## プロジェクト概要
AWS構成のポートフォリオ。IaCはTerraform、CI/CDはGitHub Actions(OIDC)。

## 運用ルール
- 1タスク完了ごとに意味のある単位でgit commitする(diffが追えるように)
- 迷ったら推測で進めず、STATUS.mdに疑問点を書いて次のタスクに進む
- IAMロールは必ず最小権限で設計する(ワイルドカードResourceの使用は避ける)

## 過去の失敗から学んだルール(随時追記)
- 2026-06-XX: OIDCのtrust policyでsub条件を書き忘れ、誰でもAssumeRoleできる状態になった。
  → IAMロール作成時は必ずtrust policyのsub条件を明示すること。
- 2026-06-XX: DynamoDBをOn-DemandではなくProvisionedで作ってコストが想定外に高くなった。
  → 個人開発規模のDynamoDBは特に指示がない限りOn-Demandを使う。
```

**解説**: MitchellHashimotoの言う「エージェントが間違いを犯すたびに、二度と同じ間違いを犯さないよう仕組みを作り込む」という考え方を体現するのが、この「過去の失敗から学んだルール」セクションです。Ghosttyのリポジトリでは、AGENTS.mdの各行が過去のエージェントの失敗1つ1つに対応しています。 タミさんの場合、AWS構成で一度事故った(想定外のコスト、権限過多など)ら、その都度一行足していくと、ポートフォリオ自体が「運用しながら育てたハーネス」の実例になります。

## レベル3: Hooks(Sensors — 事後検証)

これが差別化ポイントになります。実行後に機械的にチェックする層です。

```json
// .claude/settings.json に追加
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/check-terraform.sh"
          }
        ]
      }
    ]
  }
}
```

```bash
#!/bin/bash
# .claude/hooks/check-terraform.sh
# .tfファイルが変更されたらfmtとvalidateを自動実行し、結果をClaudeに返す

CHANGED_FILE=$(cat | jq -r '.tool_input.file_path // empty')

if [[ "$CHANGED_FILE" == *.tf ]]; then
  cd "$(dirname "$CHANGED_FILE")" || exit 0
  FMT_RESULT=$(terraform fmt -check 2>&1)
  VALIDATE_RESULT=$(terraform validate 2>&1)

  if [ -n "$FMT_RESULT" ] || echo "$VALIDATE_RESULT" | grep -q "Error"; then
    echo "{\"decision\": \"block\", \"reason\": \"terraform fmt/validateでエラー: $VALIDATE_RESULT。修正してから次に進んでください。\"}"
    exit 0
  fi
fi
exit 0
```

**解説**: HooksはPreToolUse、PostToolUseなどのライフサイクルイベント発生時にシェルスクリプトを実行できる仕組みで、提案されたアクションはJSON形式で標準入力に渡されます。 このようなセンサーは「Computational(計算的)」なsensorに分類され、決定論的で高速です。すべての変更のたびに、エージェントと並走して安価に実行できるのが利点です。 ポイントは、ただエラーを返すのではなく「LLMが消費しやすいように最適化された信号」を返すことです。自己修正の指示を含んだカスタムlinterメッセージは、一種の"良性のプロンプトインジェクション"として機能し、エージェントが自力で次に何をすべきか分かるようになります。

## レベル4: Inferential Sensors(LLM-as-judge、コスト高だが柔軟)

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/security-review.sh"
          }
        ]
      }
    ]
  }
}
```

IAMポリシーやセキュリティグループの変更時だけ、別のClaude Codeセッションを呼んでレビューさせる、という使い方です。

**解説**: Inferentialな制御はセマンティックな分析、AIによるコードレビュー、いわゆる「LLM as judge」にあたります。GPUで実行されるため遅く高価で、結果もより非決定論的になりますが、非決定論的であっても、強いモデル(タスクに適したモデル)と組み合わせれば信頼性を大きく高められます。 全ファイルに使うとコストが跳ね上がるので、IAM・ネットワーク設定など「間違えたら影響が大きい変更」に絞って使うのが実務的です。

## レベル5: Memory / Observability(前回までの内容の再整理)

- `git log`: 1タスク1コミットのルールで、後から人間が差分を追える状態にしておく

## 全体像とタミさんへの推奨ステップ

多くの開発者はレイヤー1(CLAUDE.md)とレイヤー2(Permissions)の一部までしか持っていません。 なので:

1. **今すぐ**: `deny`リストの整備(terraform destroy/apply、iam delete系)— 5分で効果大
2. **次**: CLAUDE.mdに「過去の失敗ルール」セクションを作り、事故るたびに追記する運用を始める
3. **余裕があれば**: `terraform validate`を自動で回すPostToolUseフックを1つ入れてみる

ポートフォリオのREADMEに「このプロジェクトはハーネスエンジニアリングの実践として、permission設計・失敗駆動のCLAUDE.md・自動検証フックを組み合わせている」と書けると、単なる「Claude Codeで作りました」より技術的な深みが伝わります。
