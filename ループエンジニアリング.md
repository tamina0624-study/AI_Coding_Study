「ループエンジニアリング」も最近出てきた言葉なので、最新の使われ方を確認しますね。タミさんとこれまで話してきた「一晩放置」「Agent Teams」「/goal」の話が、実はこの1つの概念に集約されます。2026年6月に急速に広まった、比較的新しい言葉です。

## 定義: プロンプトを書く人からループを設計する人へ

ループエンジニアリングとは、AIエージェントに対して1回1回自分でプロンプトを打つのではなく、スケジュールに沿ってエージェントにプロンプトを送り続けるシステムを設計する実践です。再帰的なゴールを定義し、エージェントに仕事を見つけさせ、実行させ、結果を検証させ、何が完了したかを記憶させる仕組みを作り、そのシステム自体にエージェントを動かさせます。

プロンプトエンジニアリングは、自分の手で1ターンずつ打ち込む単一の指示を最適化するものです。一方ループエンジニアリングは、何をいつプロンプトすべきか、結果が許容できるかどうかを判断する自律システムそのものを最適化します。プロンプトエンジニアリングはエージェントを「手に持つ道具」として扱いますが、ループエンジニアリングはエージェントを「記憶・スケジューリング・評価・オーケストレーションを備えた長時間稼働するプロセス」として扱います。

## 誰が言い出したか

この言葉が名前を得たのは2026年6月、Addy Osmaniがエッセイを発表し、Peter Steinbergerが「もうエージェントにプロンプトを打つのはやめて、エージェントにプロンプトを送るループを設計すべきだ」と一文に凝縮した週でした。Claude Codeを作ったBoris Chernyも同じことをベンダー側の視点で語っており、自分の仕事はもうClaudeを動かすループを書くことだと述べています。

## 元になった技術: "Ralph"

ループエンジニアリングと呼ばれる前から、この考え方の原型は存在していました。2026年初頭、Geoffrey Huntleyは単純なwhileループの中でコーディングエージェントを走らせる手法を説明しました。書かれた仕様書に対して同じプロンプトをエージェントに与え、1つのタスクを選んで実装させ、新しいインスタンスを立ち上げて同じプロンプトを再び与える、というのを完了するまで繰り返します。彼はこれをシンプソンズのキャラクター、ラルフ・ウィグムにちなんで名付けました。決定論的なほどシンプルなのに、予測不能な世界でもうまく機能するからです。

重要な洞察は「コンテキストのリセット」にあります。長時間の1セッションはウィンドウが古い推論・行き止まりの試行・古くなったファイル内容で埋まっていくと劣化していきます。 これは前回話した「フェーズごとに新しいセッションを立ち上げる」パターンと同じ発想です。

## ループを構成する4つの要素

エージェントが次に何をし、いつ止まるかを決めるのは、トリガー・トポロジー・検証者(verifier)・停止ルールの4つです。良いプロンプトは1ターンを直すだけですが、ループはその1ターンを繰り返し実行させる仕組み全体を扱います。

特に重要なのが検証者です。

どんなループでも、ボトルネックはモデルではなく検証者(verifier)です。「良い」「完了」が何を意味するかを測定可能な形で定義することが、ほとんど誰も語らない、しかし最も重要なポイントです。

## Claude Codeでの実装: /goal

これは既にタミさんが使い始めている機能です。

Claude Codeは2026年5月11日、v2.1.139で/goalをネイティブ機能として追加しました。これはターンをまたいで、自分で書いた条件が真になるまで走り続けるループで、別の高速モデルが毎ターン後に作業を採点します。

つまり`/goal`自体が「検証者が組み込まれたループ」であり、まさにループエンジニアリングの実装例です。

## タミさんの運用に当てはめる

これまで一緒に組んできたものを、この4要素で整理し直すとこうなります。

| 要素 | タミさんの実装 |
|---|---|
| トリガー | 寝る前に手動起動 → 将来的にはRoutines等でスケジュール化も可能 |
| トポロジー | Agent Teams(design/dev/reviewer)、または単一セッションでの`/goal` |
| 検証者 | `terraform validate`フック、reviewerエージェント、PostToolUseでのlintチェック |
| 停止ルール | todo.mdの全タスク完了、STATUS.mdへの記録完了 |

## Open loop vs Closed loop

オープンループかクローズドループかを選ぶという論点もあります。クローズドループは検証者の判定に基づいて自動的に再試行・修正まで行うのに対し、オープンループは1回実行して結果を人間に返すだけの設計です。タミさんの「本番デプロイは人間が確認する」というルールは、意図的に「デプロイの手前でループを開く(オープンにする)」設計だったと言えます。

## まとめると

プロンプトエンジニアリング → コンテキストエンジニアリング → ハーネスエンジニアリング → ループエンジニアリングという流れの中で、タミさんはすでに:
- **ハーネス**: CLAUDE.md、permission設定、hooks
- **ループ**: `/goal`、todo.md+STATUS.mdでの一晩運用、Agent Teamsでのdesign→dev→reviewer

を実践してきています。次に強化するとしたら「検証者(verifier)の精度を上げる」こと — つまり、reviewerエージェントやhooksが「完了」をどれだけ正確に判定できるかが、ループ全体の信頼性を決める一番のレバレッジポイントです。


実際に運用する上での注意点を、具体的な設定例とセットで解説します。

## 気を付けること1: 検証者(verifier)なきループは「速く間違える機械」になる

検証者を持たないオープンループは、予測可能かつ高コストな形で失敗します。エラーの積み重なり、幻覚的な進捗報告、静かな失敗、ゴールドリフト、そして同じ壊れたアプローチを表面だけ変えて繰り返す"doom loop"です。検証者とは、結果を期待される結果と照合する何か—コンパイラ、テストランナー、linter、diffチェック、判定役としての別モデル、あるいは人間の承認ゲートです。

**設定例**: 良い条件と悪い条件の違い

```
❌ 悪い例(検証不能)
/goal claude-md-rust.mdをプロダクション対応にする

✅ 良い例(1コマンドで検証可能)
/goal bash tools/lint-claude-md.sh examples/claude-md-rust.md が exit 0を返し、
ファイルが100〜180行の範囲内で、少なくとも3つの関連ガイドにリンクする
「See Also」セクションを含む
```

3つのチェック可能な条件があれば、Claudeはlinterを実行し、行数を数え、セクションをgrepできます。ループは数ターンで収束し、きれいに停止します。それに対し「production-readyにする」だけでは検証可能な終了状態がなく、あなたが割り込むまでトークンを消費し続けます。

## 気を付けること2: 成功条件だけでなく「上限」を必ずセットする

常に2つの停止条件が必要です。成功条件(完了したこと)と上限(最大試行回数、最大時間、最大コスト)です。成功条件だけあって上限がないループは、達成できないゴールを追いかけてトークン予算を使い果たします。

**設定例**:

```
/goal tests/api/ 配下の全テストがpassし、npm run lintがexit 0を返す。
20ターンを超えたら停止する。
```

/goalには「N回試して諦める」というビルトインの仕組みはありません。上限が欲しければ条件自体に書き込む必要があります(「...または20ターン後に停止」)。

headless実行(`-p`モード)の場合はさらに危険です。

ヘッドレスのgoalモードは最も強力であると同時に最も危険です。曖昧な条件で/goalを実行する設定ミスのCIジョブは、一晩で予算を焼き尽くします。ヘッドレスの/goalには必ずタイムアウトラッパーとトークン上限を組み合わせてください。

**設定例**(タイムアウト付きheadless実行):

```bash
timeout 3600 claude -p "/goal tests/api/配下の全テストがpassする。または20ターン後に停止。" \
  --max-tokens-per-turn 8000
```

## 気を付けること3: checker(判定役)は自分でコマンドを実行できない

重要なポイントは、各ターンの後に別の(デフォルトではHaikuの)小さいモデルがトランスクリプトを読んで「完了したか?」を判断することです。つまりコードを書いたエージェント自身が採点しているわけではありません。ただし注意すべき点があります。そのcheckerは自分でコマンドを実行しません。Claudeが既に提示した内容を判定するだけです。したがって条件は、Claude自身の出力で示せるものでなければならず、コマンドを実行しないとわからないような私的な事実ではいけません。

**設定例**(checkerが判定できる形にする):

```
❌ 悪い例
/goal 認証コードにセキュリティ上の問題がないこと

✅ 良い例
/goal tests/auth/ の全テストがpassし、`npm run lint`がクリーンであること
（Claudeにテストとlintを実行させ、その結果を出力に表示させる）
```

## 気を付けること4: 複数エージェントが同じファイルを触るとコリジョンが起きる

複数のエージェントを走らせた瞬間、ファイルが衝突し始め、それが失敗の原因になります。2つのエージェントが同じファイルに書き込むのは、2人のエンジニアが誰も相談せずに同じ行にコミットするのとまったく同じ問題です。git worktree(同じリポジトリ履歴を共有しながら独自のブランチを持つ、別の作業ディレクトリ)がこれを解決します。

**設定例**:

```bash
# Agent Teamsやパラレルタスクを使う際、各エージェントを独立したworktreeで動かす
git worktree add ../task-manager-dev feature/dev-branch
git worktree add ../task-manager-review feature/review-branch

# Claude Codeでは --worktree フラグで同様の分離が可能
claude --worktree
```

## 気を付けること5: モデル選択がコストを大きく左右する

Goalモードはターンごとのコストをそのまま増幅します。1ターンあたり5,000トークンを消費するセッションが想定より長く回ると、5,000×12ターンになります。レバーは3つ: 理由がなければSonnet 4.6を使う。30ターンのループでOpus 4.7を使うとコストがすぐ積み上がる。オーバーレイ(進捗表示)を監視し、ターン数とトークンが伸びているのに進捗が見えなければ割り込む。ループは自力では抜け出せません。

**設定例**:

```
/model sonnet
/goal ドキュメントカバレッジが完全であること
```

## 気を付けること6: 目を離す前に「観測可能」にしておく

監視されないループは、監視されないまま失敗します。一晩任せる前に観測可能にしておきましょう。出力を、後で見る場所に送ること—ログファイル、Slack/Discordメッセージ、あるいはトリアージ受信箱。すでに閉じてしまったターミナルではありません。失敗時も含め、毎回1行書き込むこと—各ステップごとにprogress.md(またはログ)にタイムスタンプ付きのメモを追記し、何を試し、何が通り、何が壊れたかを記録します。

**設定例**(タミさんが既に使っているSTATUS.mdパターンをそのまま活用):

```markdown
# progress.md への追記ルール(CLAUDE.mdに記載)

各ターンの終わりに以下を追記すること:
- [timestamp] 試したこと: ...
- [timestamp] 結果(pass/fail): ...
- [timestamp] 次にやること: ...
```

## まとめ: 今すぐ使えるチェックリスト

| 項目 | 確認内容 |
|---|---|
| 検証条件 | 1コマンドで検証可能か(vagueな形容詞が入っていないか) |
| 停止条件 | 成功条件 + ターン数/トークン/時間の上限を書いたか |
| checker適合性 | Claudeの出力(テスト結果等)で示せる条件か |
| ファイル分離 | 複数エージェントならworktreeで隔離したか |
| モデル | 必要以上に高性能なモデルを使っていないか |
| 観測可能性 | ログ・通知先を用意したか、閉じたターミナルに流していないか |

タミさんの一晩放置タスクなら、まずは`/goal`の条件を「terraform validateがexit 0、READMEにアーキテクチャ図のセクションが存在する」のような検証可能な形に書き換え、「30ターンで停止」を明示的に足すところから始めるのがおすすめです。
