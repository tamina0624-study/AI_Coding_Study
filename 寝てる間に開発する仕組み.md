## ステップ1: 寝る前にタスクをチェックリスト化する

プロジェクトのルートに `todo.md` を作り、粒度を「1項目=数分〜30分で終わる単位」まで分解します。粒度が粗いと1ターンで終わらずコンテキストを圧迫し、細かすぎると管理が煩雑になります。

```markdown
# todo.md

## CI/CDパイプライン整備
- [ ] GitHub Actions で OIDC 認証の workflow を作成 (ECR push用)
- [ ] ECS へのデプロイステップを追加し、動作確認
- [ ] 失敗時にSlack通知が飛ぶようにする

## README強化
- [ ] Enterprise Landing Zone の README にアーキテクチャ図を追加
- [ ] DevSecOps Pipeline の README に設計意図(なぜその構成にしたか)を追記
- [ ] 各READMEにデプロイ済みURLへのリンクを追加
```

## ステップ2: 進捗ファイル(STATUS.md)を用意する

コンテキストが溢れて新しいセッションに引き継ぐ場合や、`/goal` が試行錯誤する際の"作業ログ"として使います。

```markdown
# STATUS.md

## 現在のステータス
未着手

## 完了したタスク
(ここに自動追記させる)

## 失敗した試み・理由
(例: "OIDC設定でrole trust policyのconditionが厳しすぎてAssumeRole失敗。sub条件をrepo:owner/repo:ref:refs/heads/main に修正して解決")

## 次にやること
todo.md参照
```

タミさんに指示するプロンプト例(CLAUDE.mdか、セッション開始時の指示に含める):
> 「各タスクが終わったらSTATUS.mdの『完了したタスク』に追記し、詰まった場合は『失敗した試み・理由』に記録してから次の方法を試すこと。全部終わったら作業を止めてよい。」

## ステップ3: CLAUDE.mdに安全ルールを明記する

自動承認モードで一晩放置するので、事故防止のガードレールを先に書いておきます。

```markdown
# CLAUDE.md

## 一晩の自律作業ルール
- 本番環境(prod)へのデプロイ・削除系コマンドは絶対に実行しない
- AWSリソースの削除(terraform destroy含む)は行わない
- 不明な点があれば推測で進めず、STATUS.mdに疑問点を記録してそのタスクをスキップし次に進む
- 各タスク完了後は必ず動作確認(テスト実行/curlでの疎通確認など)を行う
- コミットはこまめに行い、1タスク=1コミットを目安にする
```

## ステップ4: バックグラウンドで起動する

PCをスリープさせない設定にしたうえで、`tmux` を使うのがおすすめです(セッションが独立して生き続け、朝起きたらアタッチして結果を見られるため)。

```bash
# tmuxセッションを起動
tmux new -s overnight-dev

# セッション内でClaude Codeを起動
cd ~/projects/aws-master-portfolio
claude
```

セッション内で:
```
todo.mdのタスクを上から順に、/goal で最後まで実行して。
各タスク完了ごとにSTATUS.mdを更新し、git commitすること。
```

その後 `Ctrl+B, D` でデタッチ(セッションは動き続ける)。朝は `tmux attach -t overnight-dev` で戻れます。

## ステップ5: 朝チェックする項目

- `STATUS.md` の「完了したタスク」「失敗した試み」を確認
- `git log` でコミット履歴を確認(1タスク1コミットになっていれば差分が追いやすい)
- CI/CDが絡む場合はGitHub Actionsの実行結果も見る
